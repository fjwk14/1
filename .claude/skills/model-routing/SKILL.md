# モデル使い分け(トークン節約)

## 大原則
- セッションのモデルはユーザーが `/model` で切り替える(アシスト側からは変更不可)。
- 同一セッション内でモデルを切り替えても文脈は保持される。切替コストはゼロ。
- サブエージェントは別モデルを指定できる(Agent の `model:` )が、**コールドスタートで文脈を読み直す**。小さな作業ほど割高。**大きく自己完結した塊**のときだけ使う。

## ルーティング表
| 種類 | 推奨モデル | 例 |
|---|---|---|
| 要所(設計・判断) | **Fable 5** | データモデル/RLS設計、水球統計式の正しさ、曖昧なバグ調査、UX/画面デザインの良し悪し、最終レビュー、リサーチを伴う新機能設計 |
| 通常実装 | **Sonnet 5** | 画面・サーバーアクションの実装、素直な機能追加、配線 |
| 機械作業・大量処理 | **Haiku / Sonnet** | 似たファイルの一括修正、テスト/E2Eの実行と反復、検索、セレクタ修正、定型のcommit/PR |

## 実践手順(ユーザー向け)
1. 新機能や難所に入る前に `/model claude-fable-5` にして設計・方針を固める。
2. 方針が決まったら `/model claude-sonnet-5` に落として実装。
3. 大量の反復(E2Eをグリーンにする等)は Sonnet のまま、または下記サブエージェントに委譲。
4. 最後の統括レビューだけ `/model claude-fable-5` に戻す。

## 実践手順(アシスト向け: サブエージェント委譲)
次の条件を**すべて**満たすときだけ、`Agent`(`model: "sonnet"` など)に委譲してメインの高価トークンを節約する:
- タスクが自己完結していて、成功条件が明確(例: 「この3ファイルを新UIに合わせ、全E2Eをグリーンにして報告」)。
- 反復や大きな出力(テストログ等)が予想される。
- メインの会話文脈をほとんど必要としない(必要な情報はプロンプトに書ける)。

委譲プロンプトに必ず含める: 対象ファイル、成功条件、ローカルスタックの起動コマンド、検証コマンド、コミット/PRの要否。

## やってはいけない
- 小さな修正のためにサブエージェントを乱発する(コールドスタート分だけ損)。
- 要所の判断を安いモデルに投げる(手戻りの方が高くつく)。
